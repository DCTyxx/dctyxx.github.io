一、搭建Java环境

下载JDK

下载地址：https://www.oracle.com/java/technologies/javase-jdk8-downloads.html

Java语言的三种不同技术架构

  1. Java SE（Java Standard Edition）标准版
    Java SE适用于开发桌面级应用程序。桌面级应用程序就是我们从网上下载软件的安装包，把软件安装到磁盘本地，才能使用的软件。例如：腾讯QQ、PC版微信、PC版钉钉、WPS….
  2. Java EE（Java Enterprise Edition）企业版
    适用于开发企业级应用程序（Web应用）。企业级应用程序：直接通过浏览器和网络即可使用的程序。不需要下载。例如：淘宝网、天猫、京东、唯品会、百度、新浪、网易….
  3. Java ME（Java Micro Edition）微缩版
    适用于做嵌入式设备、移动端设备的开发。例如：数字机顶盒、车载导航、手机游戏

Java三个版本的关系

Java SE定义了Java语言的语法规范。Java SE是其他另外两个版本的基础。如果想要使用Java EE或Java ME进行开发的话，一定不能够脱离Java SE。

区分JDK和JRE

JRE

JRE全称叫做Java Runtime Environment，翻译为Java运行环境。JRE中包含了运行一个Java程序所需要的最基本的类库，以及最重要的运行平台JVM（Java虚拟机）。无论在什么设备上面，只要需要运行Java程序，那么JRE就是必不可少的环境。

JDK

JDK全称叫做Java Development Kit，翻译为Java开发工具包。JDK中包含了开发一款Java程序所需要用到的一系列工具，例如：
java.exe 运行工具
javac.exe 编译工具
javap.exe 反编译工具
javadoc.exe 文档生成工具
jar.exe 打包工具
……
当然了，JDK中也包含了JRE。

什么时候应该安装JRE？什么时候应该安装JDK？

  1. 如果仅仅需要运行Java程序，那么安装JRE就够用了。
  2. 如果需要开发Java程序，必须安装JDK。

安装JDK

在MacOS下安装JDK

直接把从官网上下载好的dmg后缀的文件双击打开，双击打开pkg后缀的安装包，按照提示安装即可。

在Windows下安装JDK

1. exe后缀的可执行程序安装
第一页面，点击“下一步”。
第二页面，可以选择要安装的功能（默认即可，开发工具、源码、JRE）。
还可以更改JDK的安装路径。默认安装到C盘下某个路径，如果不想占用系统盘的空间，也可以自行修改到其他的位置。
【注意】

  1. 自定义的路径中尽量不要出现中文字符
    因为中文字符可能会因为编码而出现一些问题。
    D:\java\jdk1.8.0_241 符合规范
    E:\tools\java\jdk1.8.0_241 符合规范
    F:\学习\java环境\jdk1.8.0_241 不符合规范
    不仅仅Java，后边要用到的好多跟软件开发相关的工具，
    安装的路径中统统不要出现中文字符。

  2. 尽量选择一个固定的路径去安装，安装好之后JDK路径就尽量不要再变了
    JDK安装路径是不是绝对不能改变？
    JDK可以随便移动，挪动完之后需要重新修改环境变量即可。

  3. 要记住自己选择的安装路径，一会儿配置环境变量还要用到这个路径。
    JDK所在的根路径通常称之为“JAVA_HOME”，代表Java的安装位置。
    有很多软件都会依赖这个JAVA_HOME的值。

2. 压缩包类型的安装包
直接把压缩包解压到磁盘某个路径即可。后边好多相关工具、插件全都是这种解压+配置式的安装方法。例如：Tomcat、Maven、Nginx、MySQL数据库…..

在Windows下配置环境变量

1. 环境变量在哪里配置？
我的电脑右击–>属性–>左侧“高级系统设置”–>上方“高级”选项卡即可看到“环境变量”选项，点击打开即可。需要配置的有三项内容：

  1. 新建一个变量，变量名为JAVA_HOME
    变量值就是JDK根路径。例如：

    C:\Java\jdk1.8.0_241

  2. 新建一个变量，变量名为CLASSPATH
    变量值是一个英文小数点.（代表当前目录）

  3. 注意，不是新建变量，而是在系统中本来就存在的环境变量PATH中，添加一个变量值。
    一定不要新建一个名为PATH的变量，
    否则会将原有的PATH直接覆盖掉，
    导致CMD中有好多指令都不能使用了。
    添加的变量值是JDK根路径下面的bin目录。

    C:\Java\jdk1.8.0_241\bin

通过观察，我们会发现Path中添加的这一个变量值前面的一长串，
就和之前配好的JAVA_HOME变量值是完全相同的。
那么这时候我们就没有必要把一长串路径再重复地定义一次。
我们可以直接引用已经配置好的JAVA_HOME变量的变量值，
需要我们做的就是在这个变量值后边手动地添加一个”\bin”就可以了。
写作：

%JAVA_HOME%\bin

这样做的好处：

1.   方便书写Path的变量值，尤其是在JAVA_HOME路径较长的情况下避免了重复书写大段的路径前缀。
2.   方便频繁地修改JDK版本，如果JAVA_HOME、Path以及其他需要用到JAVA_HOME变量值的变量全部都使用绝对路径定义死，每次修改JDK都需要修改全部相关的环境变量。如果使用引用的方式，无论有多少个变量用到了JAVA_HOME，都只需要修改JAVA_HOME这一个变量即可。
3.   像这种引用的思想，在后边的编程当中会经常出现。

值得注意的是win7和win10略有不同，win7的视图中不会把PATH变量对应的所有变量值都像列表那样排列出来。

2. 三个环境变量的具体作用

  1. JAVA_HOME
    1. 方便其他的变量引用，便于修改JDK的版本
    2. “约定俗成”的变量，为了方便告诉第三方软件JDK安装的位置。后边要用到的一些工具或者插件，有一些用Java语言开发的。那么，在运行这些工具的时候，它们需要知道Java的安装路径。通常情况，它们就会去系统中寻找一个名为“JAVA_HOME”的变量值。如果找到了，而且路径真实有效，工具就可以运行。如果找不到，则不能正常运行，比如说Eclipse。
    3. 如果仅仅需要在CMD中手动执行Java相关的命令，实际上JAVA_HOME没有必要配置的，只要Path中的bin目录配置正确即可。
      注意，这时候Path中配置的bin目录必须使用真实路径。

  2. CLASSPATH
    这个变量代表类加载的路径。在Java程序中，需要用到某个类，就必须先把这个类的.class字节码文件加载到内存中。这个步骤称之为“类加载”。
    旧版本的配法：

    .;%JAVA_HOME%\lib\dt.jar;%JAVA_HOME%\lib\tools.jar

    dt.jar和tools.jar是什么东西？
    运行Java程序所需要的系统级别的类库。jdk1.5版本之后，不需要再刻意配置dt.jar和tools.jar这两个路径了。类加载器会自动到指定路径下面去加载这两个文件。实际上，这个小数点都不需要配置。jdk1.5版本之后，如果系统中没有配置CLASSPATH环境变量，那么默认的类加载路径就是当前所在路径。如果做实际开发，生产机建议保留CLASSPATH。保证向下兼容。而且还有一些工具强制性要求配置CLASSPATH变量，否则不能用。例如，在Linux平台下使用Ant。

  3. Path
    作用是为了能在CMD中执行java、javac等相关的命令。如果将来我们使用Eclipse等IDE工具编写和运行Java程序，实际上不需要在Path中配置%JAVA_HOME%\bin。

    【小案例】了解Path变量的作用
    前提：Path是Windows中的一个重要的环境变量，和Java并没有直接的关联。

    1. 打开CMD，切换到C:\Windows\System32目录。System32这个目录中包含了Windows系统中很多命令工具。

    2. 在这个路径下面执行一条命令：ping www.baidu.com
      ping命令用来测试连通性的，用法：ping ip/域名

问题：为什么能够在这里使用这条ping命令？
实际上，CMD中能够执行的每一条命令，都对应当前所在目录下的一个可执行程序。所谓的ping这条命令的执行，实际上是在执行当前路径下的一个叫做ping.exe的可执行程序。因为在C:\windows\system32这个目录中，存在一个叫做ping.exe的可执行程序，所以我们在CMD中才能使用ping这条命令。如果，有一位程序员需要频繁地测试和某一台主机的连通性，那么是否意味着，他需要每次打开CMD之后都要先执行繁琐的切换目录步骤？

cd C:\Windows\System32  --> ping --> ping.exe

磁盘中的各个位置都保存着很多我们日常需要执行的可执行程序，如果每次执行前都需要先切换到可执行程序所在的路径，也未免太过于繁琐了。 所以Windows系统提供了一个名为Path的环境变量，设定规则：只要是配置在Path变量值中的路径，那么这些路径下的可执行程序都可以随时在任意磁盘位置调用。

我们为什么要把%JAVA_HOME%\bin配置到Path的变量值中？
java当中的相关指令其实也都是对应着具体的应用程序 

java --> java.exe
javac --> javac.exe

JDK中所有的可执行文件（java.exe,javac.exe,jar.exe,javap.exe,javadoc.exe….）全部都保存在JDK根路径的bin目录下。

能够在CMD中成功执行java、javac等命令的决定因素是什么？

1.   磁盘中的某个路径中包含了两个可执行程序：java.exe、javac.exe
2.   路径配置到了Path环境变量中

【思考】如果我们安装好JDK之后没有配置任何环境变量（JAVA_HOME、CLASSPATH、Path都没有配置）是否能够在CMD中使用javac、java指令完成一个Java程序的编译和运行？请简单分析说明原因。
在控制台中将路径切换到java.exe与javac.exe路径下是，通过指令可以完成java程序的编译与运行。

检测安装和配置是否成功

CMD = Command 命令
Win+R键打开运行界面 –> 输入CMD –> 回车
如果是MacOS系统的话，在启动台打开终端分别键入下列三条指令，查看结果：

java 
javac
java -version （注意中间有个空格）

如果java指令可以正常执行，说明什么？

1.   java.exe可执行程序是存在的
2.   java.exe所在的路径已经成功被配置到了Path中，所以我们在任意位置都能用java指令
3.   Path中配置的bin目录引用了%JAVA_HOME%的路径，说明JAVA_HOME的配置也是正确的

四、JVM

JVM全称叫做Java Virtual Machine，翻译为Java虚拟机,是Java语言运行的平台。

编程语言的发展历史

第一代：机器语言 由0和1组成的二进制指令
第二代：汇编语言 指令有了英文名字 mov add jmp loop….
第三代：高级语言 C、C++、C#、Java、Python、PHP…..
第四代：SQL语言
无论是哪一种编程语言、哪一代的编程语言，最终都是要直接或间接地翻译成机器语言交给CPU执行。CPU除了0和1，别的什么都不认识。

***C语言为什么需要编译？***

C的源代码我们能看懂，但是CPU看不懂。想要交给CPU去执行，就必须得翻译成它能理解的内容（二进制指令–机器码）。

字节码、JVM和CPU的关系

Bytecode 要运行的Java代码,JVM 作用就是把16进制的字节码（Bytecode）翻译成CPU能理解的二进制指令（机器码）。
JVM在Java代码和CPU之间充当一个翻译的角色，翻译的过程就叫做解释执行。
CPU 中央处理器

Java程序完整地运行流程

1）第一步
编写.java后缀的源代码文件（source file）
问题：Java的源代码直接运行在CPU上？ —-不行
2）第二步
使用javac编译器，将.java源文件进行编译,如果语法没有错误的话，则编译成功，会生成.class为后缀的字节码文件（类文件、bytecode）生成的.class文件名和.java文件名是相同的。
问题：字节码文件能不能直接运行在CPU？ —-不行
CPU只能直接运行0和1组成的二进制机器码。
【注意】字节码和机器码不是一回事
3）第三步
直接使用java.exe程序运行Java类。这一步操作在我们看来好像是在直接运行.class文件，但是其实不是。这一步实际上是把生成的.class文件交给了Java虚拟机。由Java虚拟机把字节码逐字逐句地翻译成机器指令，交给CPU运行。这个过程，就叫解释执行过程。

编译型语言和解释型语言的区别

编程语言分为编译型语言和解释型语言。
编译型语言：
源代码不能直接运行。通常存在一个编译器，能够把一整篇的源代码一次性翻译成一整篇的机器码。而且会把翻译的结果保存在一个文件中。接下来就可以直接把这个文件交给CPU运行。典型代表：C语言对语法要求极为严格，只要有一点语法错误，整篇编译都会失败。
解释型语言：
无需编译，源代码就可以直接运行。注意，不是把源代码直接交给CPU运行。解释型语言都有对应的解释器，解释器的作用就是可以逐字逐句地读取源代码，一边读取，一边翻译成机器码交给CPU执行。边读、边翻译、边执行。不会把翻译的机器码结果保存到一个文件中。典型代表：JS对语法要求不那么严格。
【思考】Java语言属于编译型语言还是解释型语言？
Java语言属于半编译半解释型语言。
虽然java语言需要编译，但是不像C语言那样直接把源代码翻译成机器码。java编译器把java源代码翻译成字节码，字节码是不能直接交给CPU执行的。字节码需要再交给JVM解释执行。

Java到底是如何实现的跨平台？**

因为JVM是不跨平台的，所以才能让源代码和字节码跨平台。
因为SUN公司开发了很多不同版本的JVM虚拟机，分别对应不同的操作系统。不同版本的JVM能够把相同的字节码文件翻译成不同操作系统中的机器码运行。我们编写的源代码是相同的，经过javac编译生成的字节码也是相同的。不同操作系统中的JVM能够把相同的.class字节码文件翻译成对应平台不同的机器码运行。
例如：

HelloWorld.java ---javac---> HelloWorld.class

如果在Windows中运行，windows版本的JVM就能把HelloWorld.class翻译成Windows系统能看懂的机器码执行。
如果在Linux中运行，Linux版本的JVM就能把HelloWorld.class翻译成Linux系统能看懂的机器码执行。

五、GC（Garbage Collection，垃圾回收器）

什么是垃圾？ –> 垃圾内存 –> 没有用的内存

取消了内存分配

在Java中只有new操作开辟空间，没有任何的操作回收空间。内存空间不需要程序员考虑如何回收。因为有GC的存在，允许我们Java程序员编程的时候“有前手，没后手”。

Java系统线程自动管理内存

GC本质上就是一个后台进程，随着JVM的启动而启动，开始工作，工作在后台。GC会定期地关注内存的使用情况，只会在它认为有必要的时候进行内存的回收和释放。

我们可以调用下面的这两个方法【建议】GC进行垃圾回收：

java.lang.System.gc()
java.lang.Runtime.gc()

注意：仅仅是提出建议，到底会不会执行垃圾回收还是要看GC自己。我们没有任何办法强迫GC进行垃圾回收。

常见的垃圾回收算法

1. 标记清理法
“哪里脏了扫哪里”
GC会对内存中的垃圾位置进行标记，有针对性地释放这些内存单元的空间。适用于某块内存中垃圾比较少的情况。

2. 内存搬移法
找到一块内存空间中有用的内存，把这些有用的内存完整地搬移到另外的一块内存空间当中。剩下的全部都是垃圾，完整地、一次性地全部进行回收。适用于某块内存中垃圾比较多的情况。

六、代码安全验证机制

一个Java程序从诞生到最终被执行，总共分为两个大的阶段：
1. Compile 编译期

  1. 创建并编写Java源代码文件.java
  2. 使用javac编译器对源代码进行编译，生成.class（字节码文件、类文件、bytecode）
  3. 把.class文件提交给JVM准备运行
    2. Runtime 运行期
  4. 类加载器把.class文件加载到JVM内存当中
  5. 进行字节码验证
    验证内容：
    a）代码必须符合JVM规范
    b）代码不能破坏系统的完整性
    c）不能造成堆栈的上溢和下溢
    d）参数类型必须正确
    e）类型转换必须正确
  6. 把字节码交给解释器，逐条解释执行字节码中的内容
  7. 每解释一条，相当于把一句字节码翻译成了一句机器码，交给硬件设备

JIT即时编译器、热点代码编译器，Java语言为了实现跨平台特性，不可避免地要牺牲一部分性能。JVM将字节码解释执行的过程会降低Java程序的运行效率。所以出现了JIT来对运行期间进行优化，能够提高一定的运行效率。JIT的原理是在开始解释执行字节码之前，先对整个字节码文件进行扫描和探测，寻找字节码中的热点代码，对这些热点代码进行特殊处理。热点代码：在一段程序中多次重复出现的代码找到热点代码之后，JIT会把它们一次性直接编译成对应当前操作系统的机器指令批量执行。这样能够避免相同代码重复地花费时间解释执行。

package包声明

package用来声明当前源代码文件中的类所在的包，可以不出现。但是如果出现了就必须放在源文件的最前面。
实际项目开发中，往往一个Java项目中会有很多个类。如果把所有的类全部都放在一个目录下面，难免会有诸多不便。为了方便我们管理项目中的Java类，以及避免命名冲突的问题，Java中给我们提供了package包机制。使用包来整理类的主要思想把功能相似的类放在同一包下。类似于我们在系统中使用文件夹对文件进行分类。
包名的一般书写方法：公司名.小组名.项目名.组件…..;一般我们会使用公司域名倒写的形式作为包名的前缀。

原来的HelloWorld现在完整类名其实已经变成了

十三、Java中的类加载机制

类加载：当程序中需要使用到某个类的时候（无论是Java API中自带的类，还是我们自己编写的Java类），必须要先把这个类加载到内存中。其实是把保存有这个类描述信息的文件（.class）装载到JVM中。字节码文件中的所有信息会被解析读取出来，变成JVM内存中的一个Class类的对象。ClassLoader类加载器负责到指定的路径中查询.class文件，并且加载到内存中。
Java中有三种类加载器：
1）BootStrapClassLoader 引导类加载器/启动类加载器（grandpa）
主要负责加载系统中的核心类库。
类加载路径：%JAVA_HOME%\jre\lib
启动类加载器用C++编写的，用户是没有办法直接调用的。
2）ExtClassLoader 扩展类加载器 （father）
主要负责加载系统中的扩展类。
类加载路径：%JAVA_HOME%\jre\lib\ext
扩展类加载器是用Java语言编写的。
3）AppClassLoader 应用类加载器/本地类加载器 （son）
主要负责加载本地类，也就是我们自己编写的Java类。
加载路径：%CLASSPATH%
应用类加载器是使用Java语言编写的。

程序中现在出现了一个加载任务（需要用某个类了），到底是由哪个类加载器来完成这次的加载任务？
类加载机制：【双亲委托机制/双亲委派机制】
1. 三个加载之间的关系
【启动类加载器】是【扩展类加载器】的父加载器。
【扩展类加载器】是【本地类加载器】的父加载器。
2. 某个类需要被加载的过程
一个类什么时候需要被加载？
直接用java指令运行某个类，在程序中声明某个类的变量、创建某个类的对象、访问某个类中的静态变量/方法…..加载任务产生之后，首先会通知【本地类加载器】。所有的类加载器都有自己的缓存空间。缓存空间中保存的就是自己已经加载过的一些类的信息。当某个类加载器接收到加载任务通知时，都会先到自己的缓存空间中查找之前是否加载过这个类。如果缓存中存在，则直接返回使用，这一次加载任务结束。如果缓存中不存在，那么它并不会自己直接尝试加载，而是向父级加载器进行委派。
3. 例如，现在有真实的一个类HelloWorld需要进行加载
【本地类加载器】接收到加载的请求。【本地类加载器】先查询自己的缓存空间，查看该类是否被加载过（是否存在被加载过的结果 Class对象）。
.class –> Class对象
如果有，则直接拿来使用。如果没有，不会自己尝试加载，委派给父级加载器【扩展类加载器】。
【扩展类加载器】先查询自己的缓存空间，查看自己是否加载过该类。如果有则直接返回给son【本地类加载器】去使用。如果没有，则继续向上委派，委派给grandpa【启动类加载器】。
【启动类加载器】先查询自己的缓存空间，查看自己是否加载过该类。如果有则直接返回并使用。
问题：如果启动类加载器也没有加载过这个类则怎么办？
到此，向上委派的过程结束。因为启动类加载器已经是最高级别的加载器了，它没有父亲了。
【注意】向上委派的过程只是让三个加载器去尝试着从自己的缓存空间中查找要加载的这个类，之前是否加载过。因为如果有的话，就可以直接拿来用了，就不需要再重复地加载了。这个过程当中就算都没有找到加载过的记录，也没有关系。向上委派结束之后，还有一个向下加载的过程。
4. 向下加载的过程
按照从上到下的顺序，依次到自己的类加载路径中查询是否存在指定的类信息（.class文件）。如果找到，则执行加载并返回。如果没找到，则通知下级加载器让它自己尝试解决。
【启动类加载器】先尝试加载。它会到自己的类加载路径（%JAVA_HOME%\jre\lib）中查找是否存在指定的HelloWorld.class文件。如果有，则执行加载，并且返回。如果没有，告诉【扩展类加载器】：我这里没有那个类的信息，你自己想想办法。
【扩展类加载器】也会先到自己的类加载路径（%JAVA_HOME%\jre\lib\ext）中查询是否存在指定的HelloWorld.class文件。如果有，则加载这个类，并且返回。如果没有，告诉【本地类加载器】我这儿和你爷爷那儿都没有你要的类信息，你自己去找找吧。
【本地类加载器】就会到自己的类加载路径（%CLASSPATH%、执行指令的时候临时添加的-cp、-classpath所指定的路径）中查找是否存在这个HelloWorld.class。如果存在，则执行加载。可以在程序中使用这个类了。如果不存在，则抛出异常ClassNotFoundException。找不到或无法加载该类，类加载失败。
